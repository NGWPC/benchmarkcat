BenchmarkCat STAC Deployment Guide

OVERVIEW
========
Deployment of BenchmarkCat STAC Catalog on AWS for OWP:
- EC2-based STAC API (Docker Compose: PostgreSQL + pgstac, STAC API, STAC Browser, asset proxy service)
- S3 storage (~1.5 TB geospatial assets + STAC metadata)
- Cross-account S3 migration from NGWPC to OWP

ARCHITECTURE
============
- EC2 Instance: t3.xlarge (4 vCPU, 16 GB RAM)
- Services: PostgreSQL (5432), STAC API (8082), STAC Browser (8080), asset-proxy service (8083)
- Storage: S3 bucket with stac/, data/, docs/ directories
- Bootstrap: Automated via deployment/terraform/templates/user_data_standalone.sh.tpl. 
   Can be executed manually: deployment/terraform/user-data/owp-bootstrap.sh


DEPLOYMENT PHASES
=================

1. TERRAFORM EXECUTION
-------------------------------
Internal NGWPC Testing:

Infrastructure team will provision via Terraform:
- Security Group (owp-benchmarkcat-sg)
- IAM Role (owp-benchmarkcat-s3-access)
- EC2 Instance with bootstrap script

Post-Provisioning:
- Retrieve EC2 public IP
- Verify SSH access with defined key pair
- Verify bootstrap success: `cat /var/log/benchmarkcat/bootstrap.log`

Handoff to OWP:
Initial setup:
- Follow Steps in terraform/README.md

Post-Apply:
- Retrieve EC2 public IP (`terraform output`)
- Verify SSH access (`ssh -i ~/.ssh/owp-benchmarkcat-key.pem ubuntu@<ec2-ip>`)
- To Destroy (if required) : `terraform destroy`

If terraform execution is not an option, all required resources and necessary configurations
can be referenced from terraform code, and provisioned using the AWS console.

2. BOOTSTRAP SCRIPT EXECUTION (Automated - for reference)
------------------------------
Script: deployment/terraform/templates/user_data_standalone.sh.tpl

Automated Setup:
- System packages (Docker, Python, AWS CLI, PostgreSQL client)
- Directory structure (/opt/benchmarkcat, /var/log/benchmarkcat)
- Docker Compose stack (database, stac-api, stac-browser, asset-proxy)
- Management scripts (health-check, backup, restart)
- Systemd service for auto-start
- Weekly automated backups (cron)

Verification:
- /opt/benchmarkcat/deployment/health-check.sh
- curl http://localhost:8082/
- curl http://localhost:8080/


3. S3 MIGRATION (Cross-Account)
--------------------------------
Scripts:
- deployment/s3_migration/migrate_s3.py
- deployment/s3_migration/run_migration.sh

Prerequisites:
- IAM migration role provisioned (owp-benchmarkcat-migration-role)
- NGWPC bucket policy updated for cross-account access
- OWP account ID provided to NGWPC team

Migration Process:
1. Dry run preview:
   python3 migrate_s3.py --source-bucket fimc-data --dest-bucket owp-benchmark-stac --dry-run

2. Execute migration (~8-12 hours):
   python3 migrate_s3.py --source-bucket fimc-data --dest-bucket owp-benchmark-stac

3. Verify structure:
   aws s3 ls s3://owp-benchmark-stac/stac/ --recursive | wc -l  # ~22,000 files
   aws s3 ls s3://owp-benchmark-stac/data/ --recursive | wc -l

Result:
- s3://owp-benchmark-stac/stac/        (~200 MB metadata)
- s3://owp-benchmark-stac/data/        (~1.5 TB assets by collection)


4. CATALOG LOADING
------------------
Script: deployment/scripts/load_catalog.py

Prerequisites:
- Clone repository: git clone https://github.com/NGWPC/benchmarkcat.git
- Verify database ready: docker exec benchmarkcat-db pg_isready -U pgstac -d stacdb

Process:
1. Get database password
   export PGPASSWORD=$(grep POSTGRES_PASSWORD /var/log/benchmarkcat/bootstrap.log | sed 's/.*POSTGRES_PASSWORD=//')

2. Dry run (preview)
   python3 load_catalog.py /path/to/catalog --db-host localhost --db-password $PGPASSWORD --dry-run

2. Load catalog:
   python3 load_catalog.py /path/to/catalog --db-host localhost --db-password $PGPASSWORD

3. Verify:
   docker exec -i benchmarkcat-db psql -U pgstac -d stacdb -c \
     "SELECT collection, COUNT(*) FROM pgstac.items GROUP BY collection;"

Success Criteria:
- Collections load without errors
- Loading rate > 10 items/second
- All items accessible via API


5. REWRITE ASSET URLS
------------------
Script: deployment/scripts/rewrite_asset_urls.py

Prerequisites:
- Asset-proxy service running on port 8083
- Database password available (PGPASSWORD env var or /opt/benchmarkcat/.db_password)

Process:
1. Test asset-proxy functionality:
   sudo ./deployment/scripts/test_asset_proxy.sh

2. Get host IP for VPC access:
   export HOST_IP=$(hostname -I | awk '{print $1}')

3. Get database password
   export PGPASSWORD=$(grep POSTGRES_PASSWORD /var/log/benchmarkcat/bootstrap.log | sed 's/.*POSTGRES_PASSWORD=//')

4. Dry run to preview changes:
   python3 rewrite_asset_urls.py --proxy-url http://${HOST_IP}:8083 --db-host localhost --db-password $PGPASSWORD --dry-run

5. Apply URL rewrites:
   python3 rewrite_asset_urls.py --proxy-url http://${HOST_IP}:8083 --db-host localhost --db-password $PGPASSWORD

6. Verify in STAC Browser:
   Open http://${HOST_IP}:8080 and check that assets load correctly

Success Criteria:
- Asset-proxy test passes
- All S3 URLs rewritten to proxy format
- Assets render correctly in browser



5. COMPONENT TESTING
---------------------

TC2.1: Database Health
- Verify pgstac extension installed
- Check database connectivity from API container

TC2.2: STAC API Endpoints
- Test: /, /collections, /search endpoints
- Verify: HTTP 200 responses, valid GeoJSON, correct S3 HREFs

TC2.3: STAC Browser
- Access: http://<ec2-ip>:8080
- Verify: Collections display, items browsable, assets accessible

TC2.4: GDAL VSI S3 Access
- Test direct S3 asset access via GDAL Virtual File System
- Verify no authentication errors

TC2.5: Asset Proxy Service
- Test: sudo ./deployment/scripts/test_asset_proxy.sh
- Verify: Proxy returns S3 assets correctly with Range header support

Commands:
  export HOST_IP=$(hostname -I | awk '{print $1}')
  curl http://${HOST_IP}:8082/collections
  curl http://${HOST_IP}:8080/
  curl http://${HOST_IP}:8083/s3/bucket/key
  docker ps
  /opt/benchmarkcat/deployment/health-check.sh


6. POST-DEPLOYMENT VERIFICATION
--------------------------------

Health Checks:
- /opt/benchmarkcat/deployment/health-check.sh
- systemctl status benchmarkcat
- docker ps
- docker stats --no-stream

Service Access:
- STAC API: http://<ec2-ip>:8082
- STAC Browser: http://<ec2-ip>:8080
- Asset Proxy: http://<ec2-ip>:8083
- SSH: ssh -i ~/.ssh/owp-benchmarkcat-key.pem ubuntu@<ec2-ip>

Backup Verification:
- crontab -l | grep backup-db.sh
- /opt/benchmarkcat/deployment/backup-db.sh
- ls -lh /opt/backups/postgres/


PRODUCTION READINESS CHECKLIST
===============================

Infrastructure:
□ EC2 instance provisioned with correct IAM role
□ Security groups configured (SSH restricted, ports 8080/8082 accessible)
□ Elastic IP or DNS configured (optional)

Migration:
□ S3 migration completed (~1.5 TB transferred)
□ Catalog structure verified (stac/ and data/ directories)
□ Asset HREFs updated to new S3 bucket

Application:
□ Docker Compose stack running (4 containers)
□ PostgreSQL initialized with pgstac
□ STAC API responding on port 8082
□ STAC Browser accessible on port 8080
□ Asset-proxy service running on port 8083

Testing:
□ All component tests passed (TC2.1 - TC2.5)
□ Health check script passes
□ External access verified
□ GDAL VSI S3 access confirmed
□ Asset-proxy test passes
□ Asset URLs rewritten to use proxy

Operations:
□ Systemd service enabled for auto-start
□ Automated backups scheduled and tested
□ Documentation updated with actual IPs/endpoints


ROLLBACK PLAN
=============
If deployment fails:
1. Preserve logs from /var/log/benchmarkcat/
2. Infrastructure team destroys resources via Terraform
3. S3 destination bucket cleaned if needed (optional)
